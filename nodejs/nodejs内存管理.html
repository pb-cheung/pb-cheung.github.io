# V8虚拟内存管理

# 1.V8内存结构

# Heap memory（堆内存）

  • 用来存储对象（objects）和动态数据（dynamic data）。
  • 堆内存中并不是所有空间都要进行垃圾回收
  • 进行垃圾回收的只有New Space和Old space

# New Space

新空间（新生代），存储新的对象，这些对象通常存活时间较短。

新空间被分为两个空间相等的子空间(semi spaces)：from-space和to-space，它们二者是可以互相转换的。

--min_semi_space_size --max_semi_space_size

# Old Space

旧空间（老年代），存储存活时间较长的对象，新空间中经过两次minor GC的对象就会被放到旧空间来。

--initial_old_space_size --max_old_space_size

# Larget Object space

大型对象空间。存储其他内存空间都装不下的大型对象，不会被GC清理。

# Code space

存储及时编译器（JIT）编译过的代码块。有可执行内存的空间。

# Cell space、Property cell space、Map space

这些该空间包含的对象的大小都一致，简化了gc。

# Stack memory（栈内存）

  • V8每个进程分配一个栈内存空间
  • 栈存储静态数据：
    • 方法函数帧（frame）
    • 字面量（primitive values）
    • 对象的引用
  • 操作系统管理

# 2.Garbage Collection（垃圾回收）

栈内存由系统管理，无需V8引擎来处理。

堆内存占据了大量的内存空间，且存储着动态数据。如果不加以管理，它可能会指数级增长耗尽内存，或者在一段时间后碎片化拖慢我们的应用。

释放孤儿对象(orphan object)占据内存，然后重新分配给进程使用。

孤儿对象：1. 没有被栈中的元素所引用； 2. 没有被堆中的另一个对象引用

# 垃圾回收

两个阶段（stage）三种算法

# Minor GC

  • 发生在新空间（new space）
  • 触发时机，内存分配指针到到达新空间边界
  • 会阻塞程序运行，但是非常快和高效，使用并行的辅助线程，

清道夫算法scanvenger

scanvenger [ˈskævɪndʒə(r)]，用到的算法是Cheney

Minor GC 具体过程：

  1. from-space中01-06块已经被分配占用
  2. 现在进程需要创建一个新的内存块07来使用
  3. 发现From-space中内存空间不够，这样就触发了 minor GC
  4. minor GC 通过栈中的指针（GC Root）递归遍历对象树，找出仍在使用的对象，将它们移动到to-space，它们引用的对象也一并被移动过去。与此同时，因为在新的空间挨个排列，也完成了压缩减少了碎片。
  5. 重复4.的过程，直到遍历完from-space中所有的对象
  6. 清空From-space（剩余的未被引用的objects就是被清理的垃圾），垃圾清理完成，from-space和to-space角色发生互换

  1. 进程创建object10，空间不够，重复1.-6.过程

# Major GC

  • 作用域old space
  • 进行清理和压缩
  • 触发时机，V8引擎认为没有足够的old space（随着minor GC动态计算容量）

使用算法：Mark-Sweep(标记清除)和Mark-Compact（标记整理）

Mark-Seep算法：1.从GC Root递归深度优先遍历标记存活的对象，2.gc清理未被上一步标记存活的对象

Mark-compact算法：前两步和标记-清除算法一样，多了第3.步压缩整理，所有存活的对象被移动放置在一起，来减少碎片。

# 为什么不能使用minor GC阶段的Scavenger算法：

  1. 新空间存活对象少，Scanvenger算法适合，以空间换时间
  2. 旧空间存活对象多，存活时间长，如果使用Scanvernger算法则复制存活对象效率低（要复制大部分），而且要浪费一半的空间。

# 并行辅助线程

过程解释：

  1. 完成两个minor gc循环依然存活的对象，被移动到old space，old space没有足够的空间，启动major gc
  2. 多个和主线程并行的辅助线程进行标记操作
  3. 标记完成或者达到内存空间限制，主线程停止原有任务，做标记完成操作然后协调帮助线程清理和压缩、更新（指针）操作

# 最大内存

默认最大内存：

  • 32位系统中，1.5GB
  • 64位系统中，4GB

在 32 位系统中，由于内存地址的限制，一个进程最多只能使用 4GB 的虚拟内存空间，其中大约有 2GB 的空间用于操作系统和其他应用程序。因此，Node.js 默认的最大内存限制为 1.5GB。 在 64 位系统中，由于内存地址的扩展，一个进程可以使用更大的虚拟内存空间，因此 Node.js 默认的最大内存限制为 4GB。

修改方法

node --max-old-space-size=8192 server.js //单位为MB

测试方法：

# 4.内存泄露

内存泄露从常见场景原因：

a. 全局变量 b. 闭包引用 c. 事件重复监听 d. 缓存爆炸

# a.全局变量

未声明的变量（会直接绑定到全局对象Global或Window上）和全局变量下的变量，即使不再使用也不会被自动回收。

解决：delete或者赋值为null、undefined，改变之前的引用。

function test() {
  x = new Array(100000);
}

test();
console.log(x); // [ <100000 empty items> ]

# b.闭包引用

同一个作用域生成的闭包对象是被该作用域中所有下一级作用域共同持有的

var theThing = null
var replaceThing = function () {
  var originalThing = theThing
  var unused = function () {
    if (originalThing)
      console.log("hi")
  }
  theThing = {
    longStr: new Array(1000000).join('*'),
    someMethod: function () {
      console.log(someMessage)
    }
  };
};
setInterval(replaceThing, 1000)

例子来自：https://blog.meteor.com/an-interesting-kind-of-javascript-memory-leak-8b47d2e7f156

讨论：https://github.com/ElemeFE/node-interview/issues/7

解释： unused 使用了作用域的 originalThing 变量，因此 replaceThing 这一级的函数作用域中的闭包（someMethod）对象也持有了 originalThing 变量（重点：someMethod 的闭包作用域和 unused 的作用域是共享的），之间的引用关系就是 theThing 引用了 longStr 和 someMethod、someMethod 引用了 originalThing、originalThing 又引用了上次的 theThing，因此形成了链式引用。

在这个例子中，unused 函数确实是一个闭包，即使它没有被 replaceThing 函数返回。闭包指的是在一个函数内部定义的函数，这个内部函数引用了它外部函数的变量，并且这个内部函数的作用域链包括它外部函数的作用域。在这个例子中，unused 函数在 replaceThing 函数内部定义，它引用了 originalThing 变量，而 originalThing 变量又引用了 theThing 变量。因此，unused 函数可以访问 theThing 变量，并且它的作用域链中包括了 replaceThing 函数的作用域。因此，unused 函数符合闭包的定义。

# c.事件重复监听

这种情况很容易在复用对象上添加事件时出现。

node.js中HTTP keepalive模式是开启的，当客户端和服务器之间的 TCP 连接保持打开状态时，Node.js 进程会一直保持该连接的内部状态，直到连接关闭或超时。如果您的应用程序使用了 HTTP keep-alive 并且在处理请求时没有正确释放内存，那么在长时间运行的情况下，这些未释放的内存可能会逐渐增加，最终导致内存泄漏。

解决：

  1. 使用 Node.js 内置的 http.Agent 类来控制 HTTP keep-alive 的行为，可以设置 maxSockets 参数限制在特定时间内打开的 TCP 连接的最大数量。
  1. 通过减少每个 TCP 连接上的 HTTP 请求的数量或缩短 HTTP 请求之间的时间间隔来限制 HTTP keep-alive 的使用。

# d.缓存爆炸

使用内存（Object/Map）做缓存是我们开发中常见的方式。如果不注意控制缓存大小和过期时间，可能会造成失效的数据仍然在内存中，导致缓存溢出。

缓存中存储的键越多，长期存活的对象就越多。

举例：

const cache = {};

function setCache() {
  cache[Date.now()] = new Array(1000);
}

setInterval(setCache, 100);

上例中，一直添加缓存，不进行清理，会导致缓存撑爆。

解决：使用lru-cache (opens new window)组件，通过LRU淘汰算法来避免缓存爆炸。

# 内存泄露监控及定位解决

# heapdump + Chrome Devtool 定位解决

heapdump在运行初期生成一个初始内存快照文件，运行一段时间后（发生了内存泄露后）再生成一个内存快照

heapdump.writeSnapshot('init.heapsnapshot');

...

heapdump.writeSnapshot('leak.heapsnapshot');

将两个快照导Chrome 入Devtool的Memory面板进行分析。Delta标识变化的数量，Size Delta表示变化空间的大小。

通过内存快照找到数量不断增加的对象，找到增加对象是被谁给引用。

# pm2

生产环境中可以使用pm2守护进程。

设置内存消耗达到限制时的重启机制：

pm2 start index.js --max-memory-restart 8G

可以同时通过负载均衡器实现多个节点来避免服务中断。

# node-memwatch

原生模块，当它检测到正在运行的代码中存在内存泄漏时，会触发一个事件。

const memwatch = require("memwatch");

memwatch.on("leak", function (info) {
  // event emitted
  console.log(info.reason);
});

该事件通过 leak 发布，其回调对象有一个 reason 属性，会表示经过连续的垃圾回收后，堆内存占用逐渐增长。

# 如何避免内存泄露

  1. Lint工具检测代码检查非期望的全局变量。
  2. 使用闭包的时候，得知道闭包了什么对象，还有引用闭包的对象何时清除闭包。最好可以避免写出复杂的闭包，因为复杂的闭包引起的内存泄漏，如果没有打印内存快照的话，是很难看出来的。
  3. 绑定事件的时候，一定得在恰当的时候清除事件。在编写一个类的时候，推荐使用 init 函数对类的事件监听进行绑定和资源申请，然后 destroy 函数对事件和占用资源进行释放。
