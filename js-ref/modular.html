# JS模块化

将复杂的程序根据一定的规则（规范）封装成几个块（文件），并进行组合在一起使用。 块内部的数据/实现是私有的，只是向外部暴露一些接口（方法）与外部其他模块通信。

# 历史

# 1.原始模式

问题：global被污染，容易命名冲突

function foo() {}
function bar() {}

# 2.namespace模式

var MYAPP = {
    foo: function() {},
    bar: function() {}
}
  • 减少global上的变量
  • 本质是对象，不安全（使用者可以轻松更改）

# 3.匿名闭包（IIFE模式）

var Module = (function() {
    var _private = 'safe value'
    var foo = function() {
        console.log(_private)
    }
    return {
        foo: foo
    }
})()
Module.foo()
Module._private // undefined
  • 函数是JavaScript唯一的LocalScope

# 4.3基础上增强：引入依赖

var Module = (function($) {
    var _$body = $('body')
    var foo = function() {
        console.log(_$body)
    }
    return {
        foo: foo
    }
})(jQuery)
Module.foo()
  • 这就是模块模式。也是现代模块实现的基石

# 为什么要模块化

  • 降低（系统、代码）复杂度
  • 解耦
  • 方便部署

# 模块化的好处

  • 避免命名冲突
  • 更好的分离，按需加载
  • 更高复用性
  • 高可维护性

# 模块化规范

# CommonJS规范

# 规范

  • 每个文件都当做一个模块
  • 在服务器端：模块的加载是运行时同步加载的
  • 在浏览器端：模块需要提前编译打包处理（浏览器不认识require关键字）

语法：

  • 暴露模块
module.exports = {value};
exports.xxx = value;
  • 引入模块
require('xxx')
// 自定义模块xxx为模块路径
// 第三方模块xxx为模块名

# 实现

  • 服务器端实现

    Node.js

  • 浏览器端实现

    Browserify(也称为CommonJS浏览器端打包工具)

# AMD规范

# 规范

AMD——Asynchronous Module Define（异步模块定义），顾名思义是用于浏览器端的异步模块加载规范。

语法：

  • 暴露模块
// 定义有依赖的模块
define([module1, module2], function(m1, m2){
    return 模块;
})
  • 引入模块
require([module1, module2], function(m1, m2){
    // 使用m1/m2
    return 模块;
})

# 实现——Require.js

# CMD规范

浏览器端CommonJS，和cjs有很多共同特点：同步，动态加载（运行时）。

# 规范

  • 专用于浏览器端
  • 模块的加载是异步的，模块使用时才加载

语法：

  • 暴露模块
  • 引入模块
// 定义没有依赖的模块
define(function(require, exports, module){
    exports.xxx = value
    module.exports = value
})
// 定义有依赖的模块
define(function(require, exports, module){
    // 引入依赖模块（同步）
    var module2 = require('./module2')
    // 引入依赖模块（异步）
    require.async('./module3', function () {
        
    })
    // 暴露模块
    exports.xxx = value
})
