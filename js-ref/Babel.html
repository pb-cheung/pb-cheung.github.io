# Babel

# 介绍

用作语言转译（官网说是编译器-compiler是不准确的），最初用作ES6 => ES5，名字是“6to5”。

编译（compile）：高级语言 => 机器语言 转译（transpile）：高级语言 => 另一种高级语言

# 工作原理

Babel转译分为三个步骤：

Parse => Transform => Generate

# 1.Parse

解析阶段，将源码解析为AST（抽象语法树），这一步又分为两个阶段：

  1. 词法分析（Lexical Analysis），源码字符串解析为token(令牌？)
  2. 语法分析（Syntactic Analysis），根据token生成AST

词法分析示例：

// 源码
n * n

// token可以被看做是一个扁平的语法片段数组
[
  { type: { ... }, value: "n", start: 0, end: 1, loc: { ... } },
  { type: { ... }, value: "*", start: 2, end: 3, loc: { ... } },
  { type: { ... }, value: "n", start: 4, end: 5, loc: { ... } },
  ...
]

AST示例：

每一层都有相同的结构：

{
  type: "FunctionDeclaration",
  id: {...},
  params: [...],
  body: {...}
}

可以看出程序是一个表达式（EXpressionStatement），操作符是“*”，有左右两个操作数，右侧的操作数它的标识是“n”（name: "n"）

代码转换成AST，有一个在线转换工具 astexplorer (opens new window)

# 2.Transform

转换步骤，接收上一步生成的AST，对齐进行深度优先遍历，对节点进行增删更新等操作，插件主要在这个阶段工作。

# 3.Generate

代码生成，将上一步处理过的AST，深度优先遍历，生成代码字符串。

# 配置

# preset

预设preset可以理解为一组plugin和配置的集合。

常见的preset有：@babel/preset-env、@babel/preset-react、@babel/preset-typescript

# @babel/preset-env

它的作用是根据目标环境（浏览器、Node.js 等）和配置的浏览器兼容性需求，自动确定需要进行转译的语法和功能，并使用相应的插件进行转译。它能根据目标环境的支持情况，自动选择合适的转译插件和配置，以尽可能地保持代码的原始语法特性，并最小化转译后的代码大小。

目标环境通过targets字段配置：

{
  "targets": "> 0.25%, not dead"
}
// or
{
  "targets": {
    "chrome": "58",
    "ie": "11"
  }
}

它的原理是通过维护一个映射来实现，映射可以理解为是这样的：

{
    JS某个语法特性：对应的babel插件,
    浏览器某个特性：对应的polyfill
}

# polyfill

# 一、为什么要用polyfill

叫做垫片，可以理解为兼容性补丁，通过引入它来解决环境中缺少的API等问题。

新的语法特性是有区别的：

  1. let/const、箭头函数等高级语法
  2. Promise 这样的全局内置对象
  3. 静态方法Array.from Object.assign，实例方法Array.prototype.include
  4. generator函数

只有1.可以通过转译的方法解决（所以被称为语法糖？），其他的都需要引入polyfill才能使用。

Babel中使用的polyfill有core-js和regenerator runtime。

例如：

// 源代码
function addAll() {
  return Array.from(arguments).reduce((a, b) => a + b);
}

// 转译后的代码
function addAll() {
  return Array.from(arguments).reduce(function(a, b) {
    return a + b;
  });
}

上例中，箭头函数被转为普通函数声明。但是转译后的代码运行可能还会有问题，运行环境可能不支持Array.from。为了解决这个问题就需要polyfill了。

# 二、如何配置使用polyfill

  1. 未使用webpack，直接在入口文件require('@babel/polyfill')或者import "@babel/polyfill"（不推荐）

  2. 使用webpack，未使用@babel/preset-env，修改webpack配置，入口文件前引入（不推荐）

module.exports = {
  entry: ["@babel/polyfill", "./app/js"],
};
  1. webpack + @babel/preset-env使用。推荐！

具体是通过preset-env的useBuiltIns选项来控制使用polyfill。

  • false: 默认值，不使用，如果入口引入则是全量引入polyfill。
  • entry: 需要手动引入@babel/polyfill，根据对目标浏览器版本的支持(browserlist或targets配置)，将polyfill拆分引入浏览器不支持的polyfill。这样会导致实际用不到的polyfill也会被打包到输出文件，导致文件比较大。
  • useage: 不需要手动引入@babel/polyfill，根据对目标浏览器版本的支持(browserlist或targets配置)+代码中实际使用到的新API按需添加polyfill。

# 运行时优化

# 作用

  • @babel/runtime+@babel/plugin-transform-runtime，使用这样一个组合是为了通过代码复用（babel内置的helpers）的方式达到缩减代码体积的目的。
  • 次要目的是避免引入的polyfill污染全局变量。

简单的引入polyfill一般业务场景不会有问题，但是开发给别人使用的公共组件就不合适。

# 官方示例

未使用runtime优化的代码：

// 源码
class Circle {}

// 转换后代码
function _classCallCheck(instance, Constructor) {
  //...
}
var Circle = function Circle() {
  _classCallCheck(this, Circle);
};

每处适用class的代码都会引入_classCallCheck这个函数，造成重复

适用runtime优化后：

var _classCallCheck = require("@babel/runtime/helpers/classCallCheck");

var Circle = function Circle() {
  _classCallCheck(this, Circle);
};

不再直接引入_classCallCheck，而是指向@babel/runtime中的对应代码，这个包内包含了所需函数模块化的实现。

# 最佳实践

使用@babel/preset的useBuiltIns:usage设置按需引入polyfill + @babel/plugin-transform-runtime提取重复代码的helper代码。这样生成的代码体积较小。

配置示例：

{
    "plugins": [
        [
            "@babel/plugin-transform-runtime",
            {
                "corejs": false,
                "helpers": true,
                "regenerator": false
            }
        ]
    ],
    "presets": [
        [
            "@babel/preset-env",
            {
                "corejs": 3,
                "useBuiltIns": "usage"
            }
        ]
    ]
}
