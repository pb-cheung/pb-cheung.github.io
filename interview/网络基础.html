# 网络基础

# OSI七层模型

  • 应用层：直接向用户提供服务
    • HTTP/FTP/SSH/DNS
  • 表示层：对应用层的数据进行解释，
  • 会话层：两个实体建立连接的方法，组织和协调会话进程间的通信
  • 传输层：向用户提供可靠的端到端的流量控制和差错校验
    • TCP/UDP
    • 传输连接管理：面向连接、面向无连接
  • 网络层：将报文通过通信网络传递给下一个网络节点
    • 寻址
    • 交换
    • 路由
    • 服务连接
  • 数据链路层：节点之间链路上数据的稳定准确
    • 将信息调整为语序且无差错
  • 物理层：实现计算机节点bit流传输
    • 光缆、电缆
    • 信号调制、信道复用

# TCP/IP协议

  • 应用层
  • 传输层
  • 网络层
  • 链路层

关系： TCP/IP是OSI的实现

# TCP/UDP

  1. 是否连接：TCP是面向连接的（一对一）、UDP不面向连接（一对一、一对多、广播）
  2. 是否可靠：TCP可靠（三次握手、四次回收）；UDP不可靠；拥塞控制，流量控制
  3. 传送方式：TCP:bit流 UDP:报文
  4. 开销：TCP:20字节；UDP:8字节
  5. 场景：TCP:高可靠性场景；UDP:视频流、实时性

# TCP连接建立和关闭

# 名词概念

syn： Synchronize /ˈsɪŋkrənaɪz/ Sequence Number 同步序列号 ack： Acknowledge Character 确认字符

# 为什么要了解连接建立和关闭的过程

  • 防止可能的SYN flood攻击（大量发起syn，耗尽服务端内存资源）
  • 借鉴这种思想（TLS分享传递密钥）

# TCP连接建立——三次握手

# 为什么是三次，两次能解决吗？

解释：

已经失效的连接请求报文突然又传送到服务端，产生错误。

一段时间之前，client发送了一个请求连接的报文，由于网络原因延迟很久才到达server， 如果此时server回复确认报文连接建立，client端并没有建立连接的需求，就不会理睬， server端为连接分配的内存就会浪费。

# TFO (TCP Fast Open)

TFO能实现两次握手建立连接

一、建立过程：

  1. client发送带有Fast Open选项的SYN包
  2. server生成一个cookie，回复一个SYN-ACK，并带上这个cookie
  3. client端存储cookie

二、使用过程：

  1. client发送带有cookie的Fast Open选项的SYN包
  2. server验证cookie，有效的话就回一个SYN-ACK（此时已可以发送响应数据）

# 四次挥手

# 最后client为什么要等待2MSL才进入CLOSED状态？

确定server收到client的确认应答了（未收到的话，server等待超时会重新发送断开链接的请求）

# TCP攻击

  • 半链接：client发一个SYN包，server回复一个SYN-ACK，进入SYN_RCVD状态
  • 全连接：三次握手完成，ESTABLISHED状态
