# webpack

# 作用

# 1.模块打包

不同模块的文件打包到一起，保证引用正确，执行有序。

# 2.编译兼容

loader机制实现

  • polyfill
  • SASS、LESS、Vue、JSX编译转换为浏览器可识别的CSS、JS

# 3.能力拓展

通过plugin实现

按需加载、代码压缩等

# loader机制

# 作用：

webpack默认会把所有文件都当做js进行解析。 非js类型的文件，要先对其进行转换再继续执行打包任务。这就是loader机制的作用。

# 配置

一般是通过正则匹配文件后缀，支持对同类型文件以数组形式配置多个规则。 webpack在转换时候，会链式调用每一个loader，前一个的输出作为后一个的输入。

// webpack.config.js
module.exports = {
  // ...other config
  module: {
    rules: [
      {
        test: /^your-regExp$/,
        use: [
          {
             loader: 'loader-name-A',
          }, 
          {
             loader: 'loader-name-B',
          }
        ]
      },
    ]
  }
}

# 开发

loader开发规范，返回值必须是标准的JS字符串。遵循"单一职责"原则。

# plugin

loader负责转译，插件负责功能扩展

# 工作原理

webpack基于发布订阅机制，在运行的生命周期中会广播出许多事件，插件监听这些事件， 在特定的阶段执行自己的插件任务，从而实现功能。

# 常用的插件

  • uglifyjs-webpack-plugin：压缩
  • html-webpack-plugin：将打包后的js和css静态资源插入到页面模板中
  • clean-webpack-plugin：打包前将目录清空
  • SplitChunksPlugin：分包，提升加载性能

# 开发

  • 插件必须是一个函数或者是包含apply方法的对象，这样才能访问compiler实例
  • 传给每个插件的compiler和compilation对象是同一个引用，修改会影响后续插件
  • 异步事件在处理完之后，需要回调通知webpack进入下一个流程
class MyPlugin {
  apply (compiler) {
    // 找到合适的事件钩子，实现自己的插件功能
    compiler.hooks.emit.tap('MyPlugin', compilation => {
        // compilation: 当前打包构建流程的上下文
        console.log(compilation);
        
        // do something...
    })
  }
}

# 代码分离

懒加载（lazy loading）也叫按需加载，即不是一开始就加载，在需要的地方才加载。 这样

代码分离能能减小代码包体积，加快初始加载速度。常见的实现方式有以下几种：

# 一、动态导入

ES语法中的模块导入方法除了常见的import命令导入，还有import()函数动态加载方式。

使用示例：

import('./myModule.js')
.then(myModule => {
  console.log(myModule.default);
});

# webpack中使用

webpack会将使用import()函数动态导入的模块，代码分离到单独的bundle中。在浏览器运行到这行代码时，就会加载（jsonp）这个文件。

require.ensure()是webpack特有的，已被import()取代。

官方示例：

// index.js
function component() {
  function getComponent() {
    return import('lodash')
      .then(({
        default: _
      }) => {
        const element = document.createElement('div');

        element.innerHTML = _.join(['Hello', 'webpack'], ' ');

        return element;
      })
      .catch((error) => 'An error occurred while loading the component');
  }

  getComponent().then((component) => {
    document.body.appendChild(component);
});

运行打包命令后得到两个文件： vendors-node_modules_lodash_lodash_js.bundle.js 549 KiB index.bundle.js 13.5 KiB

# 二、SplitChunksPlugin插件

官方推荐配置及具体字段含义：

module.exports = {
  // ...其他配置
  optimization: {
    splitChunks: {
      chunks: 'all', // 指定拆分哪些代码块，可选值为 'all', 'async', 'initial'
      minSize: 30000, // 设置代码块的最小大小，超过该值才会进行拆分
      maxSize: 0, // 设置代码块的最大大小，超过该值将会进行拆分
      minChunks: 1, // 设置一个模块被引用多少次以上才会被拆分
      maxAsyncRequests: 5, // 设置按需加载的并行请求数的最大值
      maxInitialRequests: 3, // 设置初始页面加载时的并行请求数的最大值
      automaticNameDelimiter: '~', // 自定义拆分后的代码块命名的分隔符
      name: true, // 根据拆分后的代码块命名规则自动生成名称
      cacheGroups: {
        vendor: {
          test: /[\\/]node_modules[\\/]/, // 指定要拆分的模块路径
          priority: -10 // 设置优先级，决定拆分到哪个代码块中
        },
        default: {
          minChunks: 2,
          priority: -20,
          reuseExistingChunk: true // 设置是否重用已有的代码块
        }
      }
    }
  }
};

chunks: 指定拆分哪些代码块。常用的选项有：

  • all: 拆分所有代码块，包括同步和异步代码块。
  • async: 只拆分异步代码块。
  • initial: 只拆分初始页面加载时的同步代码块。

实践中配置示例：

module.exports = {
  // ...其他配置

  optimization: {
    splitChunks: {
      cacheGroups: {
        echarts: {
          name: 'chunk-echarts',
          priority: 20,
          test: /[\\/]node_modules[\\/]_?echarts(.*)/
        },
        vendor: {
          test: /[\\/]node_modules[\\/]/,
          priority: 10
        },
        components: {
          test: /[\\/]src[\\/]components[\\/]/,
          priority: 5
        },
        commons: {
          name: 'chunk-commons',
          minChunks: 2,
          priority: 5,
          chunks: 'initial',
          reuseExistingChunk: true
        }
      }
    }
  }
};

解释：

  • 分离出较大的第三方组件库echarts
  • 利用priority的相对大小值来控制代码拆分的优先级，上例中，先检查模块是否匹配echarts、vendor然后是components最后是common。

webpack4之前使用CommonsChunkPlugin插件做分离任务。

开源社区插件mini-css-extract-plugin: 用于将 CSS 从主应用程序中分离。

# 三、外部扩展(Externals)

通过配置将依赖的第三方库排除，不打包进构建产物中，在页面模板中单独引入（CDN）这些库。

webpack.config.js:

module.exports = {
  //...
  externals: {
    jquery: 'jQuery',
  },
};

index.html:

<script
  src="https://code.jquery.com/jquery-3.1.0.js"
  integrity="sha256-slogkvB1K3VOkzAI8QITxV3VzpOnkeNVsKvtkYLMjfk="
  crossorigin="anonymous"
></script>

解释：

  • 依赖的jQuery库文件几乎是不会修改它的，改动少=>分包
  • jQuery库是广泛使用的，很大可能用户在访问本应用之前访问其他应用的时候，已经加载缓存这个库的脚本了，使用CDN可以利用用户客户端已有的缓存
  • 可靠性要求高的业务谨慎使用：第三方CDN的稳定性，因为依赖关系需要确保依赖加载完成等

# 懒加载

# 一、基础用法

官方示例：

// print.js
console.log(
  'The print.js module has loaded! See the network tab in dev tools...'
);
export default () => {
  console.log('Button Clicked: Here\'s "some text"!');
};

// index.js
import _ from 'lodash';
function component() {
	const element = document.createElement('div');
	const button = document.createElement('button');
	const br = document.createElement('br');
	button.innerHTML = 'Click me and look at the console!';
	element.innerHTML = _.join(['Hello', 'webpack'], ' ');
	element.appendChild(br);
	element.appendChild(button);
	// Note that because a network request is involved, some indication
	// of loading would need to be shown in a production-level site/app.
	button.onclick = e => import( /* webpackChunkName: "print" */ './print').then(module => {
		const print = module.default;

		print();
	});
	return element;
}

document.body.appendChild(component());

打包结果，print模块会被单独分成一个文件：print.bundle.js、index.bundle.js

# webpackChunkName注释语法和ES6动态导入的区别

import(/* webpackChunkName: "lodash" */ 'lodash')：这种写法使用了 webpack提供的注释语法来指定代码块的名称。Webpack将会根据指定的名称将模块单独打包成一个代码块（chunk），并在需要的时候按需加载。

import('lodash')：这种写法不使用任何注释语法。它不会为该代码块指定特定的名称，而是使用默认的命名方式生成代码块文件，比如0.js、1.js等。

# 二、Vue.js中的实现

路由配置中，修改路由导入方式

// 将
// import UserDetails from './views/UserDetails.vue'
// 替换成
const UserDetails = () => import('./views/UserDetails.vue')

const router = createRouter({
  // ...
  routes: [{ path: '/users/:id', component: UserDetails }],
})

使用具名的代码块chunk（注释语法），将某个路由下的所有组件打包到同个组件中

// 三个组件都在名称为“group-user”的代码块
const UserDetails = () =>
  import(/* webpackChunkName: "group-user" */ './UserDetails.vue')
const UserDashboard = () =>
  import(/* webpackChunkName: "group-user" */ './UserDashboard.vue')
const UserProfileEdit = () =>
  import(/* webpackChunkName: "group-user" */ './UserProfileEdit.vue')

let router = new Router({
    routes: [
        {
            path:'/login',
            name:'login',
            component: import(/* webpackChunkName: "login" */
                `@/views/login.vue`)
        },
        {
            path:'/index',
            name:'index',
            component: import(/* webpackChunkName: "index" */
                `@/views/index.vue`)
        },
        {
            path:'/detail',
            name:'detail',
            component: import(/* webpackChunkName: "detail" */
                `@/views/detail.vue`)
        }
    ]
})

# 缓存

# 文件名哈希

打包生成包含hash值的文件名，以方便浏览器缓存和应用（增量）更新。

共有三种：fullhash、chunkhash、contenthash

  • fullhash，项目全量hash，打包出来的文件名完全一致，改动任何一处源码，打包产物文件的文件名全部更新

  • chunkhash，根据不同的入口文件(entry)进行依赖文件解析、构建对应的chunk，生成对应的哈希值。当某个文件内容发生变动时，再次执行打包，只有该文件以及依赖该文件的文件的打包结果hash值会发生改变

  • contenthash，只有当自己文件的内容发生变化时，打包对应的产物的hash值才变化，常用语CSS文件。

一般使用chunkhash+contenthash，js使用chunkhash，css使用contenthash。统一使用chunkhash可能会出现非预期变化（css文件本身没有变化，但是依赖其的文件变化导致其打包产物hash值一块变化）。 一方面，可以最大程度地利用浏览器缓存机制，提升用户的体验；另一方面，合理利用 hash 也减少了 webpack 再次打包所要处理的文件数量，提升了打包速度。
