# 长列表优化

大量数据项以列表形式展示是经常遇到的业务场景。

一次性直接渲染大量的列表项可能会较长时间阻塞主线程，造成页面一段时间的不可交互，大量的DOM会消耗较高的内存，造成页面卡顿。

# 一次性渲染

一次性插入所有节点渲染耗时的测定方法：

// 记录任务开始时间
let now = Date.now();
// 插入十万条数据
const total = 100000;
// 获取容器
let ul = document.getElementById('container');
// 将数据插入容器中
for (let i = 0; i < total; i++) {
    let li = document.createElement('li');
    li.innerText = Math.floor(Math.random() * total)
    ul.appendChild(li);
}

console.log('JS运行时间：',Date.now() - now);
setTimeout(()=>{
  console.log('总运行时间：',Date.now() - now);
},0)
// print: JS运行时间： 187
// print: 总运行时间： 2844
  • 第一个console代码是同步代码执行耗时
  • 第二个console是在setTimeout回调函数中，会被放入到宏任务队列中，会在下一次循环中执行，执行的时候经历过了一次渲染，可以认为是渲染的耗时

从分而治之的思路寻找解决方案

# 零、分页加载

对长列表空间维度进行拆分，分成较短的等长的“页”，通过点击按钮或者触发到每页的头尾来触发加载前后页，保持每次都渲染列表很少的一部分。

这个方式比较适合PC端应用，但是在移动端体验不够好（点击按钮和触底后再加载新内容，体验上不够丝滑）。

# 一、时间分片

时间维度进行拆分，进行延迟渲染。根据事件循环原理（每次微任务之后都要检查进行渲染），可以使用定时器setTimeout将一个长列表放到多个事件循环中，分多次进行渲染。

# requestAnimationFrame

requestAnimationFrame(fn)代替setTimeout(fn, 0)

  1. setTimeout(fn, 0)理想情况下的最快执行间隔是4ms： 大部分浏览器都有一个最小定时器间隔（minimum timer interval）的机制来限制setTimeout的最低延迟时间，一般是4ms。

  2. 屏幕刷新频率。电脑屏幕一般为60HZ => 帧率FPS(Frame per second):60 frame/s=>1000ms/60=16.7ms渲染一次，帧率受分辨率、CPU、GPU、内存的影响是实时变动的，不会稳定为60。

因为以上两点，setTimeout回调函数中的渲染可能与屏幕刷新步调不一致，就会体现为卡顿或白屏。requestAnimationFrameAPI接受一个回调函数在下次渲染前（before next repaint）调用，这样就能保持节点添加和渲染保持步调一致。

# createDocumentFragment

DocumentFragment是一个轻量级的文档对象，它是Document接口的一个子接口。它表示一个没有父节点的文档片段。 在DOM中，通常需要对文档进行修改和操作，而这些操作可能涉及到多个节点的插入、删除或替换。如果直接在文档中进行这些操作，会导致多次重排和重绘，性能会受到影响。DocumentFragment的作用就是为了解决这个问题。 可以看作是一个虚拟的容器，类似于一个临时的文档片段。你可以在DocumentFragment中创建和操作节点，然后将整个DocumentFragment一次性地插入到文档中，从而减少了对文档的多次操作。

多次触发（回流or重绘） => 只触发一次

实现代码：

//需要插入的容器
let ul = document.getElementById('container');
// 插入十万条数据
let total = 100000;
// 一次插入 20 条
let once = 20;
//总页数
let page = total/once
//每条记录的索引
let index = 0;
//循环加载数据
function loop(curTotal,curIndex){
    if(curTotal <= 0){
        return false;
    }
    //每页多少条
    let pageCount = Math.min(curTotal , once);
    window.requestAnimationFrame(function(){
        let fragment = document.createDocumentFragment();
        for(let i = 0; i < pageCount; i++){
            let li = document.createElement('li');
            li.innerText = curIndex + i + ' : ' + ~~(Math.random() * total)
            fragment.appendChild(li)
        }
        ul.appendChild(fragment)
        loop(curTotal - pageCount,curIndex + pageCount)
    })
}
loop(total,index);

# 二、虚拟列表

单个列表项内的节点较多或列表项总数较多的情况下时间分片还是会有性能问题。 Chrome的performance面板中以下几个渲染步骤时长都会变长，Recalculate Style、Layout、Update Layer Tree。

虚拟列表（Virtual List）是应对这种场景的解决方案。

# 原理

部分渲染，只渲染可视区域或者某个指定容器的区域内的列表元素。这样实际渲染的列表项数量就大大的减少了。

实现思路：

  1. 初始化容器和列表长度（根据列表项高度和数量）
  2. 监听列表的滚动
  3. 根据滚动距离计算和列表项的高度计算可视区域内列表项的起始和结束索引
  4. 根据滚动距离计算偏移量，修改对应的的样式移动列表位置transform: translate3d()

# 列表项动态高度

最佳实践是：给列表项一个初始值先行渲染，渲染完成后获取真实高度，再同步修改获取可视区域首尾两个索引值

updated(){
  let nodes = this.$refs.items;
  nodes.forEach((node)=>{
    let rect = node.getBoundingClientRect();
    let height = rect.height;
    let index = +node.id.slice(1)
    let oldHeight = this.positions[index].height;
    let dValue = oldHeight - height;
    //存在差值
    if(dValue){
      this.positions[index].bottom = this.positions[index].bottom - dValue;
      this.positions[index].height = height;
      for(let k = index + 1;k<this.positions.length; k++){
        this.positions[k].top = this.positions[k-1].bottom;
        this.positions[k].bottom = this.positions[k].bottom - dValue;
      }
    }
  })
}
