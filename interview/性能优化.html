# 性能优化

# 常见指标

指标|web.dev (opens new window)

# 一、FP:首次绘制（First Paint）

浏览器从请求内容到首次像屏幕绘制像素点（下图的Painting）的过程的时间。可以理解为时白屏时间。

获取方法：

// 此方法如果执行时机过早可能会报错
window.performance.getEntriesByType('paint')[0].startTime

// 更推荐使用 Promise 封装获取
const observerWithPromise = new Promise<PerformanceObserverEntryList>((resolve, reject) => {
  new PerformanceObserver(resolve).observe({
    entryTypes: ['paint'],
  });
});
    
observerWithPromise
  .then(entryList => {
    return entryList.getEntries().filter(entry => {
      return entry.name === 'first-paint';
    })[0];
  })
  .then(entry => console.log(entry.startTime));

# 二、FCP:首次内容绘制（First Contentful Paint）

浏览器渲染出第一个内容的时间，内容可以是文本、图片等。

FCP >= FP

获取方法：

// 此方法如果执行过早可能会报错
window.performance.getEntriesByType('paint')[1].startTime
// 更推荐使用 Promise 封装获取
const observerWithPromise = new Promise<PerformanceObserverEntryList>((resolve, reject) => {
  new PerformanceObserver(resolve).observe({
    entryTypes: ['paint'],
  });
});
    
observerWithPromise
  .then(entryList => {
    return entryList.getEntries().filter(entry => {
      return entry.name === 'first-contentful-paint';
    })[0];
  })
  .then(entry => console.log(entry.startTime));

MDN | PerformancePaintTiming (opens new window)

# 三、FMP:首次有意义渲染（First Meaningful Paint）

这个概念没有标准化定义，不同业务的关键元素不同，例如博客网站关键元素是文字，视频网站关键元素是播放器。不能使用API直接获取，现在被LCP替代了。

# 四、LCP:最大内容绘制（Largest Contentful Paint）

可视区域内（浏览器窗口）最大的可见元素出现在屏幕上的时间，核心指标。

const observerWithPromise = new Promise<PerformanceObserverEntryList>((resolve, reject) => {
    new PerformanceObserver(resolve).observe({
      entryTypes: ['largest-contentful-paint'],
    });
  });
  observerWithPromise
    .then(entryList => {
      const entries = entryList.getEntries();
      const lastEntry = entries[entries.length - 1];
      console.log(lastEntry.startTime);
    });

# 五、TTI:用户可交互时间（Time to Interactive）

用户可以与应用进行交互的时间，一般指domready事件的时间，我们通常会绑定这个事件开始写业务逻辑。

TTI 的测算建议使用 Lighthouse。

# 六、TTFB:网络请求耗时（Time To First Byte）

请求发出开始（输入地址敲下回车），到收到响应的第一个字节的时间。

TTFB 测量startTime和responseStart之间的时间损耗。

获取方式：

window.performance.timing;

const observerWithPromise = new Promise<PerformanceObserverEntryList>((resolve, reject) => {
    new PerformanceObserver(resolve).observe({
      entryTypes: ['navigation'],
    });
  });
  observerWithPromise
    .then(entryList => {
      const [pageNav] = entryList.getEntriesByType('navigation');
      console.log(pageNav);
    });

# 各个属性时间含义：

  • startTime：开始时间，0
  • unloadEventStart：前一个页面卸载开始
  • unloadEventEnd：前一个页面卸载完成
  • redirectStart：重定向开始
  • redirectEnd：重定向完成，没有发生则为0
  • fetchStart：浏览器准备好了使用HTTP请求来获取文档的时间点
  • domianLookupStart：域名查询开始时间。如果使用了持续连接或者缓存，取值等于fetchStart
  • domianLookupStart：域名查询完成时间戳
  • connectStart：开始建立连接，如果使用了持久链接，值等于fetchStart
  • SecureConnectStart：开始建立安全连接（https）
  • connectEnd：连接建立完成
  • requestStart：（向服务器）发送请求时的时间戳
  • responseStart：浏览器从服务器、缓存或者本地资源接收到响应的第一个字节之时的 Unix毫秒时间戳
  • responseEnd：接受响应的最后一个字节的时间戳
  • domInterActive：HTML解析完成，DOM树构建完成的时间戳
  • domContentLoadedEventStart：domContentLoaded发出之前，开始解析内联脚本

    DOMContentLoaded 事件的触发时机早于其他资源（如图片、样式表和外部脚本）的加载完成，因此它通常被用于执行需要操作 DOM 元素的 JavaScript 代码。通过将代码放置在 DOMContentLoaded 事件的监听器中，可以确保代码只在 DOM 准备就绪后执行，避免了访问尚未加载的元素而导致错误。

  • domContentLoadedEventEnd：解析内联脚本结束
  • domComplete：Document.readyState变为complete的时间戳
  • loadEventStart：onload事件触发的时间点
  • loadEventEnd：所有事件都执行完

# 优化

分清维度（线索），有的放矢

页面打开全流程，可以像上一节timing API中的属性那样，分为不同阶段维度进行优化。

# 一、I/O（network）维度

# 1.HttpCache阶段

  • 主资源——html页面

使用Cache-Control: no-cache，每次都会向服务器发送请求检查缓存是否有效，有效返回304去使用缓存，使其始终为最新。

一般不能使用缓存破坏（caching bust）来更新，因为URL不变。

>缓存破坏就是通过创建新文件且修改文件名，且文件名包含hash、版本号或者添加query时间戳参数等方法，建立新的缓存文件不再使用旧的文件。
  • 子资源——js、css、图片等

使用协商缓存(max-age/ETag)+缓存破坏。

脚本、字体、图片缓存一般缓存在内存（memory cache）中，非脚本的例如css文件会缓存在磁盘（disk cache）中。

常见的缓存模式|MDN (opens new window)

# 2.DNS阶段

应用用到的域名收敛，能少就少，一般两个：业务域名+CDN域名。

# 3.TCP阶段

# Keep-Alive模式

非KeepAlive模式时，每个请求/应答客户和服务器都要新建一个连接，完成 之后立即断开连接（HTTP协议为无连接的协议）；当使用Keep-Alive模式（又称持久连接、连接重用）时，Keep-Alive功能使客户端到服 务器端的连接持续有效，当出现对服务器的后继请求时，Keep-Alive功能避免了建立或者重新建立连接。

如何开启使用

http1.0中默认关闭。 http1.1中默认启用， 默认情况下所在HTTP1.1中所有连接都被保持，除非在请求头或响应头中指明要关闭：Connection: Close。

a. 浏览器发起请求时header中添加Connection: Keep-Alive

b. 服务端收到请求看到包含次字段，response中添加同样的字段，像这样：

HTTP/1.1 200 OK
Connection: Keep-Alive
Content-Encoding: gzip
Content-Type: text/html; charset=utf-8
Date: Thu, 11 Aug 2016 15:23:13 GMT
Keep-Alive: timeout=5, max=1000
Last-Modified: Mon, 25 Jul 2016 04:32:39 GMT
Server: Apache

(body)

//解释：`timeout`字段过期时间5秒（对应httpd.conf里的参数是：KeepAliveTimeout），`max`字段是最多一百次请求，强制断掉连接，

c. 这样客户端和服务端就建立了HTTP持久连接，客户端再发请求就可以直接使用这个连接了。

# 4.REQ/RES阶段

a. 缩小包体积

  • 压缩。uglify、OptimizeCSSAssetsPlugin
  • runtime。修改babel配置，babel引入polyfill优化
  • tree shaking。
  • 图片格式，如何进行分解？webp？（64KB） base64格式，体积增大1/3，不用base64则增加一个HTTP请求

b. 首屏加载的内容拆解

  • 代码分离。webpack的codeSplitPlugin
  • 懒加载。ES动态加载，Vue动态路由、React动态加载

c. 代码分离和浏览器并发请求数限制的平衡

  • 代码分离要适度。根据路由和功能拆分，减少首次加载资源的体积的同时防止拆分的过于零碎。
  • 按需加载。动态导入、异步加载。
  • 使用HTTP2/HTTP3。更高级的并发能力，一个TCP连接上的多路复用技术。

# 5.Processing阶段

拿到主文档，开始按顺序解析； 常见的外部依赖有CSS、JavaScript文件，它们的加载执行会阻塞主流程； 所以常见的优化有：

  • 使用骨架屏，生成页面骨架结构模板放入页面模板，缩短白屏时间（FP）；
  • js脚本放到</body>body闭合标签的位置，目的是为了防止脚本的加载执行影响主线程解析渲染dom；
  • 数据分析、广告等脚本添加async属性。这类脚本的特点：异步加载、加载完就执行、不关心dom。<script src="analytics.js" async></script>；
  • 其他相对独立脚本添加defer属性。特点：异步加载、文档解析完成后执行（DOMContentLoaded之前）。

# 骨架屏

SPA应用页面模板一般是空白的，需要加载运行框架js代码后将内容挂载到根节点上，这样的机制就会造成白屏时间过长。 骨架屏优化解决这一问题，在项目打包阶段将骨架内容直接放到html页面模板中（js渲染代码之前），这样在页面渲染阶段在真实页面渲染前不至于显示白屏，显示页面骨架作为过度， 缩短页面白屏时间，提升用户体验和留存度。

vue-skeleton-webpack-plugin插件配置：

// vue.config.js
const SkeletonWebpackPlugin = require("vue-skeleton-webpack-plugin");
module.exports = {
   configureWebpack: {
      plugins: [
       new SkeletonWebpackPlugin({
        // 实例化插件对象
        webpackConfig: {
          entry: {
            app: path.join(__dirname, './src/skeleton.js') // 引入骨架屏入口文件
          }
        },
        minimize: true, // SPA 下是否需要压缩注入 HTML 的 JS 代码
        quiet: true, // 在服务端渲染时是否需要输出信息到控制台
        router: {
          mode: 'hash', // 路由模式
          routes: [
            // 不同页面可以配置不同骨架屏
            // 对应路径所需要的骨架屏组件id，id的定义在入口文件内
            { path: /^\/home(?:\/)?/i, skeletonId: 'homeSkeleton' },
            { path: /^\/detail(?:\/)?/i, skeletonId: 'detailSkeleton' }
          ]
        }
      })        
      ]
   }
}

// skeleton.js
import Vue from "vue";
// 引入对应的骨架屏页面
import homeSkeleton from "./views/homeSkeleton";
import detailSkeleton from "./views/detailSkeleton";
export default new Vue({
    components: {
        homeSkeleton,
        detailSkeleton,
    },
    template: `
    <div>
      <homeSkeleton id="homeSkeleton" style="display:none;" />
      <detailSkeleton id="detailSkeleton" style="display:none;" />
    </div>
  `,
});

# 二、渲染维度

原则：尽量少的触发浏览器的回流（reflow）和重绘（repaint）

浏览器回流和重绘

# 1.避免回流和重绘

避免措施：

  • 避免使用table布局
  • 避免设置多层内联样式
  • 避免频繁修改样式，最好一次性重写style属性，将样式列表定义为class，一次性修改class属性
  • 尽可能在DOM树的最末端（树的叶子）修改改变class
  • 避免使用CSS表达式（calc）
  • 有动画的元素将其脱离文档流（positioin: absolute|fixed），这样修改样式只引起重绘，不会造成回流
  • transform代替top
  • CSS3硬件加速（GPU加速）

# 2.使用DocumentFragment

也算是1.

DocumentFragment是一个轻量级的文档对象，它是Document接口的一个子接口。它表示一个没有父节点的文档片段。 在DOM中，通常需要对文档进行修改和操作，而这些操作可能涉及到多个节点的插入、删除或替换。如果直接在文档中进行这些操作，会导致多次重排和重绘，性能会受到影响。DocumentFragment的作用就是为了解决这个问题。 可以看作是一个虚拟的容器，类似于一个临时的文档片段。你可以在DocumentFragment中创建和操作节点，然后将整个DocumentFragment一次性地插入到文档中，从而减少了对文档的多次操作。

多次触发（回流or重绘） => 只触发一次

官方示例：

const list = document.querySelector('#list')
const fruits = ['Apple', 'Orange', 'Banana', 'Melon']

const fragment = new DocumentFragment()

fruits.forEach((fruit) => {
  const li = document.createElement('li')
  li.textContent = fruit
  fragment.appendChild(li)
})

list.appendChild(fragment)

# 三、内存维度（运行时）

  • 长列表优化，分时渲染，虚拟列表
  • 计算密集型任务使用Web Worker。

Web Worker是一种在Web应用程序中使用的浏览器特性，它允许在后台线程中执行JavaScript代码，而不会阻塞主线程的执行。主线程通常用于处理用户交互和更新界面，而Web Worker可以用于执行一些计算密集型或耗时的任务，以避免影响用户体验。 Web Worker的工作原理是将JavaScript代码运行在独立的线程中，这个线程与主线程相互独立，彼此之间不会阻塞。这样，可以同时执行多个任务，提高应用程序的性能和响应速度。
